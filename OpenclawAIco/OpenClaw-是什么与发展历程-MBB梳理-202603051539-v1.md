## 0. Executive Summary（结论先行｜金字塔顶）

- **一句话定义**：**OpenClaw 是一个你在自己设备上运行的个人 AI 助手**，通过你日常使用的消息应用（如 WhatsApp/Telegram/Slack/Discord 等）与之交互；其核心是一个本地常驻的 **Gateway（网关控制平面）**，负责会话、路由、工具与事件管理。
- **本质差异**：与“只在对话框里给建议”的聊天机器人不同，OpenClaw 强调“能做事”（工具/浏览器/节点/定时任务/技能），并强调“你的机器、你的规则、你的数据控制权”。
- **架构要点（可解释其爆发的工程原因）**：采用 **单一网关 + 多渠道接入 + 多代理路由 + 工具/技能生态** 的组合，使其在“个人AI基础设施”定位上具备可扩展性和可迁移性。
- **发展节奏（可验证）**：使用日历化版本（CalVer，形如 `vYYYY.M.D`），在 2026 年 2–3 月高频发布，且安全相关更新密度极高（从 release notes 与 `CHANGELOG.md` 可见）。
- **风险与争议同样是“核心信息”**：由于其拥有真实执行能力与渠道接入能力，OpenClaw 官方把安全作为第一优先级：推出 Trust 计划、明确单用户信任模型边界、对技能市场引入 VirusTotal 扫描，并提供 `SECURITY.md` 的披露范围与报告流程。

## 1. 定义与边界（What is it / What it isn’t｜MECE）

### 1.1 OpenClaw（本报告主对象）到底是什么？

- **官方定义（README 口径）**：
  - OpenClaw 是你在自己的设备上运行的 **personal AI assistant**；
  - “Gateway is just the control plane — the product is the assistant.”（网关是控制平面，产品是助手本身）；
  - 支持大量消息渠道，并提供语音与实时画布（Canvas）。

- **官方文档口径（Docs）**：OpenClaw 是一个自托管网关，把你喜欢的聊天应用连接到 AI 编码代理/个人助手，并强调“你自己的硬件，你的规则”。

- **开源与运行环境**：MIT License；运行时要求 **Node ≥ 22**。

### 1.2 它不是什么？（避免研究跑偏）

- **不是**一个多租户、对抗性用户共享的“强隔离平台”——官方在 `SECURITY.md` 明确：OpenClaw 的信任模型并不把单一网关当作多租户安全边界。
- **不是**“数据永远不出本地”的魔法盒：状态（配置/工作区/会话）主要在本地，但模型推理仍可能通过第三方模型提供商 API、消息平台也会存储消息数据（FAQ 明确提醒）。

### 1.3 同名混淆排除：另一个“OpenClaw”（非本报告对象）

业界/开源社区同时存在 **OpenClaw（C++ 游戏重实现）**：
- 例如 GitHub 仓库 `KingWang6/OpenClaw`：对 1997 年平台游戏《Captain Claw》的重实现。

本报告所有“OpenClaw”默认指 **AI 助手/AI Agent 平台**，不指游戏项目。

## 2. 关键价值主张拆解（MBB：客户价值→产品机制→护城河）

### 2.1 目标用户与典型需求（Who｜Jobs To Be Done）

从官方定位与能力集合推断，OpenClaw 的“主战场”是：
- **个人/高级用户**：希望 24/7 有一个能在后台持续工作、可远程触达（通过消息渠道）的助手。
- **开发者/自动化重度用户**：希望助手具备工具调用、脚本执行、浏览器控制、多会话/多代理路由等能力。

### 2.2 价值主张（What｜Value Proposition）

- **渠道即入口**：不用切换到“新App”，用你已有的 WhatsApp/Telegram/Slack/Discord…来“召唤助手”。
- **本地优先与可控性**：把“个人AI”从 SaaS 形态拉回到“你自己的基础设施”（你控制数据、密钥、部署位置与规则）。
- **行动能力**：通过工具（browser/canvas/nodes/cron/skills 等）让 AI 从“建议”变为“执行”。

### 2.3 “为什么它看起来像基础设施而不是应用？”（How｜Mechanism）

官方架构文档强调：
- 每台主机运行一个长生命周期的 **Gateway 守护进程**；
- 网关是唯一拥有并打开外部消息会话的组件；
- 控制平面客户端（CLI、macOS App、Web UI）通过 WebSocket 与网关通信；
- 节点（iOS/Android/macOS node）也通过 WebSocket 连接，暴露设备能力。

这一套结构让 OpenClaw 更像“个人 AI 的控制平面/中枢”，而不只是某个聊天界面。

## 3. 产品与技术架构（MBB：从“大图”到“关键部件”）

### 3.1 顶层架构（One slide view）

- **入口层（Channels）**：WhatsApp/Telegram/Slack/Discord/… 作为用户交互入口。
- **控制平面（Gateway）**：统一 WebSocket/HTTP 端口（默认 `127.0.0.1:18789`），管理会话、路由、工具、事件、认证与配对。
- **执行表面（Tools / Nodes / Sandbox）**：
  - Tools：browser/canvas/cron/webhooks/…
  - Nodes：移动端/桌面端节点，提供相机、屏幕录制、定位等设备能力
  - Sandbox：可将非主会话放入 Docker 沙箱，降低风险半径（安全文档建议）。

### 3.2 关键机制（Why it works）

- **多代理路由（Multi-agent routing）**：把不同渠道/账户/peer 路由到隔离的 agent/workspace，避免“一个助手污染所有场景”。
- **技能市场（ClawHub）**：公共技能注册表，技能本质是包含 `SKILL.md` 的文件夹；支持搜索/安装/更新/发布与版本历史。
- **浏览器控制（Browser tool）**：提供专用的 openclaw-managed 浏览器 profile，与个人浏览器隔离；并提供 SSRF 策略、沙箱/主机控制边界等安全配置。

## 6. 风险版图（Risk Radar｜MBB：把“不可忽视”说清楚）

> 重要：官方自己把 OpenClaw 定位为“高权限、真实渠道、可执行”的系统，因此风险不是附录，而是核心信息。

### 6.1 风险类型（MECE）

- **输入操纵（Prompt Injection）**：不受信任网页/邮件/附件中的指令可诱导工具型代理执行危险动作。
- **权限过宽（High-privilege tools）**：执行命令、读写文件、浏览器自动化等具有真实破坏性。
- **渠道攻击面（Messaging surface attack surface）**：把攻击面延伸到 IM 平台（恶意私信/群聊/伪造事件等）。
- **供应链风险（Skills/Plugins）**：技能市场与插件机制容易成为投毒入口。
- **部署暴露（Internet exposure）**：错误绑定/反向代理/令牌管理不当可能导致网关暴露。

### 6.2 官方对风险的“边界声明”（重要）

- `SECURITY.md` 明确：OpenClaw 的信任模型假设是“单用户/受信任操作员边界”；把网关暴露在公网、共享给互不信任用户，不属于其可承诺的多租户安全边界。
- Trust 页面与安全文档明确：安全是纵深防御（threat model → roadmap → audit → disclosure process），并提供漏洞报告渠道与响应 SLA。

### 6.3 官方建议的核心缓解策略（可操作）

- **默认使用配对/allowlist**：DM 默认 `pairing`，未知发送者不处理消息。
- **工具最小化**：优先使用 `tools.profile: "messaging"`；对不受信任内容使用“只读/无工具”代理。
- **沙箱隔离**：建议使用 Docker 沙箱模式隔离非主会话。
- **不要公开暴露网关**：保持 `loopback` 绑定；远程访问优先 SSH/Tailscale。
- **技能与插件慎用**：只安装可信来源技能；官方已引入 VirusTotal 扫描作为一层保障，但明确不是银弹。
